1부: Git 로컬 3대 영역 전체 아키텍처 맵Slide 02
데이터의 상태 변화와 물리적 이동 흐름 이해
[Working Directory] ------- git add -------> [Staging Area] ------- git commit -------> [Local Repo]
(수정 중인 날것의 파일) (index / 커밋 대기 상태) (.git 폴더 내부 스냅샷)
    ▲                                                                         │
    └─────────────────────────────── git checkout / switch ────────────────────────────────┘

1. Working Directory

실제 코드를 치는 공간입니다. Git이 지켜보는 Tracked 상태와, 새로 만들어 모르는 Untracked 상태가 있습니다.

2. Staging Area

다음 커밋에 포함할 변경사항만 골라 담는 바구니입니다. 내부적으로 .git/index 파일로 보관됩니다.

3. Local Repository

내 컴퓨터 안의 안전한 코드 금고입니다. git commit을 하면 해시값 기반의 버전이 생성됩니다.

1부: Git 명령어 작동 원리 (add, commit, status)Slide 03
단순 암기가 아닌 내부 데이터의 이동 분석
  • git status: Working Directory와 Staging Area의 상태를 비교하여 변경된 파일을 알려줍니다.
  • git add: 파일을 Staging Area로 보냅니다. 이때 파일은 Git 내부의 압축 포맷인 Blob 객체로 바뀝니다.
  • git commit: 대기실에 있는 구조를 트리 객체로 묶어 영구적인 스냅샷을 생성하고 포인터를 전진시킵니다.
$ git status (상태 확인) -> $ git add app.controller.ts -> $ git commit -m "feat: core logic"

Untracked, Staged, Modified의 전환 눈으로 확인하기

실습 단계:

  1. 프로젝트 폴더에 test.txt 파일을 새로 만듭니다.

  4. 파일 내용을 수정하고 상태를 보면 두 영역에 동시에 존재하는 모습을 보게 됩니다.

1부: 원격 저장소 동기화와 Push 거절(Rejected)Slide 05
다수가 동시에 메인 브랜치에 Push할 때 생기는 문제
  • Remote Repository: GitHub 같이 온라인에 존재하는 협업용 중앙 저장소입니다.
  • Push 거절 원인: 내 PC의 코드 베이스가 원격 저장소의 최신 커밋보다 옛날 버전일 때 발생합니다.
  • Git의 보호 원칙: 동료가 먼저 올려놓은 코드를 실수로 지워버리는 가로채기 사고를 막기 위해 서버가 요청을 차단합니다.

경고 메시지: ! [rejected] main -> main (fetch first)

1부: 안전한 독립 작업 공간 - 브랜치(Branch) 제어Slide 07
왜 브랜치를 나눠야 하며 어떻게 안전하게 이동하는가?
  • 브랜치의 필요성: 배포 가능한 무결점 코드 줄기(main)를 보호하고, 각자 독립적인 기능 개발을 안전하게 진행하기 위해 브랜치라는 공간을 만듭니다.
  • 핵심 명령어 상세 해제:
    • git checkout -b feature/login: 브랜치 생성과 이동을 동시에 수행하는 하이브리드 명령어입니다.
    • 생성 프로세스: 반드시 main으로 이동 후 pull을 받아 최신 상태를 확보한 상태에서 새로운 브랜치를 생성해야 꼬이지 않습니다.

1부: 브랜치 운영 전략 - Git-Flow vs GitHub-FlowSlide 09
팀의 규모와 배포 주기에 따른 최적의 전략 선택

1. Git-Flow (대규모 / 정기 배포)

5개의 브랜치(main, release, develop, feature, hotfix)를 엄격하게 운영합니다.

  • 제품이 나가는 main과 개발팀 통합 허브인 develop이 분리됩니다.
  • 중간에 완충지대(release)가 있어 QA 단계가 긴 프로젝트에 적합합니다.

2. GitHub-Flow (스타트업 / 수시 배포)

main과 feature 브랜치 단 2개만 사용하는 단순한 구조입니다.

  • 기능 개발이 끝나고 PR이 승인되어 main에 합쳐지면 그 즉시 자동으로 상용 서버에 배포됩니다.
  • 속도가 생명인 웹 서비스나 스타트업에 적합합니다.
[실습 4] PR 충돌(Conflict) 상황 로컬에서 해결하기Slide 10
동일 라인 수정으로 충돌했을 때 마커 해제법

GitHub에서 충돌 경고가 뜨면 당황하지 말고 내 로컬 컴퓨터로 최신 main을 가져와 합쳐야(Merge) 합니다.

git checkout main
git pull origin main
git checkout feature/my-profile
git merge main # 여기서 CONFLICT 발생!

에디터 충돌 마커 정리

소스코드에 나타난 <<<<<<< HEAD와 동료의 코드 중 살릴 부분을 고르고 저장합니다.

git add . && git commit -m "fix: resolve" && git push origin feature/my-profile
1부: 백엔드 실무 위기대응 명령어 (Stash, Cherry-Pick)Slide 11
급박한 실무 환경에서 코드를 구출하는 테크닉

1. git stash (임시 은닉)

코드가 미완성이라 커밋하기 민망한데 급하게 메인 브랜치로 가서 버그를 고쳐야 할 때 씁니다.

git stash # 현재 작업 주머니에 숨기기
git checkout main # 메인 이동 후 업무 처리
git checkout feature/my-profile
git stash pop # 숨겼던 코드 꺼내서 복구

2. git cherry-pick (체리픽)

다른 사람이 만든 수많은 커밋 내역 중에서 특정 커밋 1개(예: 해시 a1b2c3d)만 복사해서 내 브랜치에 붙여 넣습니다.

git cherry-pick a1b2c3d
1부: 깨끗한 히스토리 관리와 버그 되돌리기 (Rebase, Revert)Slide 12
협업 가독성을 높이고 안전하게 버그를 취소하는 방법

3. git rebase (베이스 재배치)

지저분하게 얽힌 머지 커밋 자국을 없애고, 내 브랜치의 출발점을 최신 main 끝단으로 옮겨 한 줄로 이쁘게 만듭니다.

git checkout feature/my-profile
git rebase main

4. git revert (안전한 커밋 취소)

이미 push를 해버려서 과거 이력을 지우면(reset) 안 될 때, 과거 실수를 정반대로 상쇄하는 취소 커밋을 위에 새로 얹습니다.

git revert c4d5e6f (실수했던 커밋 해시)

3부: 디버깅 - console.log를 버려야 하는 이유Slide 17
주니어의 흔한 실수와 운영 환경에서의 치명적인 결함
  • 서버 마비 위험: Node.js는 싱글 스레드로 작동합니다. console.log는 동기식 출력 장치라 사용자 요청이 몰릴 때 콘솔 출력이 밀리면 서버 전체가 멈춰버립니다.
  • 정보의 휘발성: 클라우드 도커 환경에서는 서버가 재부팅되면 콘솔에 찍혔던 에러 로그가 흔적도 없이 사라집니다.
  • 올바른 대안: Winston 같은 전문 비동기 로거를 사용하거나, 소스코드 자체를 정지시켜 들여다보는 브레이크포인트(중단점) 디버깅을 배워야 합니다.

프론트엔드가 보낸 문자열과 백엔드가 원하는 숫자의 충돌
  • 장애 현상: 쇼핑몰 장바구니에 담은 상품 개수가 화면에 NaN(Not a Number)으로 깨져서 나옵니다.
  • 원인 분석: 백엔드 API는 상품 수량을 숫자인 100으로 보냈는데, 프론트엔드가 이를 문자열로 취급하여 계산을 시도하다 발생한 연산 오류입니다.
  • 해결 미션: 크롬 개발자 도구의 Network 탭 ➔ Response 창을 스캔하여 날아온 원본 JSON 데이터 타입을 확인하세요. 그 후 리액트 소스코드에서 Number(data.count) 처리를 더해 계산 오류를 격파하세요.
